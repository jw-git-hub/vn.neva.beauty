# Промт 1 — Поисковая оптимизация (SEO)

> Как использовать: вставь этот промт в Claude Code, замени строку `Страница для оптимизации` на нужный URL и запусти. Прогони так каждую страницу сайта. Технические ключи schema.org / HTML остаются на английском (это стандарт), весь пользовательский контент — на русском.

---

**Роль:** Ты — SEO-инженер. Оптимизируешь ОДНУ страницу статического сайта косметологической клиники **Neva Beauty** для русскоязычных экспатов в **Дананге (Вьетнам)**. Сайт собирается статически и публикуется на **GitHub Pages**: нет бэкенда, нет server-side rendering в рантайме, нет серверных редиректов. Вся оптимизация должна работать в статическом HTML (встроенный JSON-LD, статические метатеги, статический контент).

**Страница для оптимизации:** `[ВСТАВЬ URL, например http://localhost:8080/services/laser]`

## Жёсткие ограничения (нарушать нельзя)

1. **НЕ меняй цены** на услуги — ни в текстах, ни в разметке. В schema.org цену переноси ровно как на странице (валюта VND).
2. **НЕ выдумывай факты:** отзывы, рейтинги, число процедур, сертификаты, имена и квалификацию врачей, награды. Если данных нет на сайте и я их не дал — НЕ добавляй `Review` / `AggregateRating` и подобное. Фейковая разметка отзывов = санкции Google.
3. **НЕ меняй URL** страницы и существующие пути. На GitHub Pages нет серверных редиректов — смена URL порвёт ссылки и обнулит накопленный вес. Если URL плохой, только предложи в отчёте, но не меняй.
4. Всё, что генерируешь для пользователя (title, description, alt, тексты, содержимое JSON-LD), — **на русском**. Технические ключи/атрибуты — по стандарту (английский).

## Шаг 1. Анализ

Открой страницу, изучи её текущее состояние: title, meta, заголовки, семантику, изображения, контактные данные (NAP: имя, адрес, телефон), внутренние ссылки, наличие schema.org. Определи, о какой услуге/разделе страница и какой у неё поисковый интент.

## Шаг 2. Метатеги `<head>`

- **title:** 50–60 символов. Формула: `[услуга/тема] + Дананг + Neva Beauty` (если влезает). Главный ключ ближе к началу. Уникальный для каждой страницы.
- **meta description:** 140–160 символов. Живое описание с ключом, пользой и мягким призывом. Уникальное.
- **canonical:** `<link rel="canonical" href="абсолютный URL этой страницы">`.
- **hreflang:** `<link rel="alternate" hreflang="ru" href="...">` для страницы + `hreflang="x-default"`. Сайт русскоязычный, аудитория — русские во Вьетнаме.
- **Open Graph:** `og:title`, `og:description`, `og:image`, `og:url`, `og:type`, `og:locale=ru_RU`.
- **Twitter Card:** `summary_large_image`.
- Проверь `<meta name="viewport">` (mobile-first) и `<html lang="ru">`.

## Шаг 3. Структура и семантика

- Ровно один `<h1>` — с главным ключом страницы.
- Логичная иерархия `H2 → H3` без пропусков уровней; заголовки осмысленные (не «Услуга 1»).
- Семантические теги HTML5: `<header>`, `<nav>`, `<main>`, `<section>`/`<article>`, `<footer>`. Основной контент — в `<main>`.
- Хлебные крошки (визуальные + разметка ниже), если есть вложенность.

## Шаг 4. Structured Data (JSON-LD, встроить в HTML)

Собери валидный JSON-LD в `<script type="application/ld+json">`. Для бренда используй один связный граф. Типы под клинику красоты:

- **LocalBusiness** (точнее — `HealthAndBeautyBusiness` / `BeautySalon`; для мед. процедур допустим `MedicalBusiness`):
  - `name`, `url`, `image`, `logo`, `telephone`, `priceRange`
  - `address` → `PostalAddress` (`streetAddress`, `addressLocality`: Da Nang, `addressCountry`: VN)
  - `geo` → `GeoCoordinates` (`latitude`, `longitude`) — реальные координаты салона
  - `openingHoursSpecification` — реальные часы работы
  - `sameAs` — ссылки на реальные соцсети (Instagram, Facebook, Telegram)
  - `areaServed` — Da Nang
- **Service** для профильной услуги страницы: `name`, `description`, `provider` (ссылка на бизнес), `areaServed`. Если на странице есть цена — вложи **Offer** с `price` и `priceCurrency` **ровно как на сайте**, без изменений.
- **BreadcrumbList** — если есть вложенность.
- **Organization** — для главной/бренда.
- **FAQPage** — ТОЛЬКО если на странице реально есть вопросы-ответы (или я их дал). Не выдумывай.
- **AggregateRating / Review** — ТОЛЬКО при наличии реальных отзывов. Иначе пропусти.

**NAP-консистентность:** имя, адрес, телефон идентичны во всех местах (footer, контакты, schema). Нашёл расхождения — приведи к одному виду и отметь в отчёте. Если реальных адреса/телефона/координат нет и я их не давал — поставь заметный `TODO`-плейсхолдер и спроси меня, не выдумывай.

## Шаг 5. Изображения

- `alt` на русском, осмысленный, с ключом где уместно (не «image1»).
- `width`/`height` у `<img>` (против layout shift / CLS).
- `loading="lazy"` для картинок ниже первого экрана; главное изображение экрана — без lazy.
- Тяжёлый формат (jpg/png) — предложи WebP в отчёте (менять файлы можно, если это в моём репо и не ломает сборку).

## Шаг 6. Внутренние ссылки и техфайлы

- Добавь релевантные внутренние ссылки на связанные услуги/разделы с осмысленным анкором (не «тут»).
- Проверь/предложи в отчёте: `sitemap.xml` (генерировать при сборке) и `robots.txt` (разрешить индексацию, указать `Sitemap:`).

## Шаг 7. E-E-A-T / YMYL (важно для медицины/красоты)

Косметология — это YMYL. Усиливай сигналы доверия там, где для этого есть **реальные** основания: явные контакты, физический адрес, карта; квалификация/лицензии — только если реально есть; ссылки на соцсети и отзывы на внешних площадках. Ничего не выдумывай.

## Формат отчёта (в конце, кратко)

1. О чём страница и целевой ключ.
2. Таблица «было → стало» по title, description, H1.
3. Что добавлено в schema.org (типы).
4. Список правок по изображениям / ссылкам / семантике.
5. `TODO`/вопросы ко мне (недостающие данные: адрес, телефон, координаты, часы, соцсети, отзывы).
